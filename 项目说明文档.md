# CRM 客户关系管理系统 — 项目说明文档

---

## 一、系统设计思路

### 1.1 整体架构

系统采用**前后端分离**的经典 B/S 架构，前端 SPA 与后端 API 服务独立部署：

```
┌──────────────────────┐    HTTP / JSON REST    ┌──────────────────────┐    SQLAlchemy ORM    ┌──────────┐
│   React 18 SPA       │ ──────────────────────▶│   FastAPI 0.115      │ ────────────────────▶│  SQLite  │
│   :5173（开发）       │                        │   :8000              │                      │  crm.db  │
│   Ant Design 5       │ ◀──────────────────────│   Pydantic v2        │ ◀────────────────────│          │
└──────────────────────┘                        └──────────────────────┘                      └──────────┘
```

### 1.2 模块划分

系统按业务领域划分为三个核心模块，对应 CRM 的三个核心场景：**谁是客户 → 怎么跟进 → 效果如何**。

| 模块 | 职责 | 前端页面 | 后端路由前缀 |
|------|------|----------|-------------|
| 客户管理 | 客户信息 CRUD + 联系人管理 | `CustomerList` / `CustomerDetail` | `/api/customers` `/api/contacts` |
| 销售流程管理 | 销售机会 CRUD + 阶段推进 + 跟进记录 | `OpportunityList` / `OpportunityDetail` | `/api/opportunities` `/api/followups` |
| 数据分析 | 统计总览 + 多维可视化图表 | `Dashboard` | `/api/analytics/*` |

### 1.3 关键设计决策

- 三个模块相互独立，通过外键关联（客户 → 机会 → 跟进），各自维护，避免跨模块耦合
- 前后端均按模块拆分文件（`models/schemas/routers` 与 `pages/components`），新增模块只需增加对应文件，不改动已有逻辑
- 刻意裁减了权限管理、审批流、消息通知等功能，以保证 MVP 核心流程完整可用

## 二、数据模型设计

### 2.1 实体关系图

```
Customer（客户）
├── 1:N → Contact（联系人）
└── 1:N → Opportunity（销售机会）
              └── 1:N → FollowUp（跟进记录）
```

四个实体构成树形层级：客户是根节点，联系人和销售机会挂载于客户，跟进记录挂载于机会。通过 `CASCADE DELETE` 保证数据完整性——删除客户时自动清理其所有子数据。

### 2.2 核心实体字段

**Customer（客户）**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | Integer | PK, autoincrement | 主键 |
| name | String(100) | NOT NULL | 客户名称 |
| industry | String(50) | NOT NULL | 行业：互联网 / 金融 / 制造 / 教育 / 医疗 / 其他 |
| scale | String(20) | NOT NULL | 规模：小型 / 中型 / 大型 / 集团 |
| source | String(50) | NOT NULL | 来源：官网 / 转介绍 / 广告 / 展会 / 电话 / 其他 |
| status | String(20) | NOT NULL, default="潜在" | 状态：潜在 / 活跃 / 成交 / 流失 |
| region | String(100) | nullable | 地区 |
| address | String(200) | nullable | 详细地址 |
| remark | Text | nullable | 备注 |
| created_at / updated_at | DateTime | server_default / onupdate | 时间戳，自动维护 |

**Contact（联系人）**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | Integer | PK | 主键 |
| customer_id | Integer | FK → customers.id | 关联客户 |
| name | String(50) | NOT NULL | 姓名 |
| position | String(50) | nullable | 职位 |
| phone | String(20) | nullable | 电话 |
| email | String(100) | nullable | 邮箱 |
| is_primary | Boolean | default=False | 是否主要联系人 |

**Opportunity（销售机会）**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | Integer | PK | 主键 |
| customer_id | Integer | FK → customers.id | 关联客户 |
| title | String(200) | NOT NULL | 机会名称 |
| stage | String(20) | NOT NULL, default="初步接触" | 阶段：初步接触 → 需求确认 → 方案报价 → 商务谈判 → 赢单 / 输单 |
| amount | Float | default=0 | 预估金额（元）|
| expected_close_date | Date | nullable | 预计成交日期 |
| priority | String(10) | NOT NULL, default="中" | 优先级：高 / 中 / 低 |
| remark | Text | nullable | 备注 |
| created_at / updated_at | DateTime | 自动维护 | 时间戳 |

**FollowUp（跟进记录）**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | Integer | PK | 主键 |
| opportunity_id | Integer | FK → opportunities.id | 关联机会 |
| type | String(20) | NOT NULL | 跟进方式：电话 / 邮件 / 拜访 / 会议 / 其他 |
| content | Text | NOT NULL | 跟进内容 |
| next_plan | Text | nullable | 下一步计划 |
| created_at | DateTime | server_default | 记录时间 |

### 2.3 级联删除规则

SQLAlchemy `relationship` 配置 `cascade="all, delete-orphan"`：

- 删除 Customer → 自动删除其所有 Contact + Opportunity（及 Opportunity 下的全部 FollowUp）
- 删除 Opportunity → 自动删除其所有 FollowUp

## 三、技术选型说明

### 3.1 前端技术栈

| 技术 | 版本 | 选型理由 |
|------|------|----------|
| React | 18 | 主流组件化框架，TypeScript 支持完善，生态最成熟 |
| TypeScript | 6.x | 类型安全，编译阶段捕获接口字段不匹配等错误，可维护性高 |
| Ant Design | 5 | 企业级后台 UI 库，`Table / Form / Drawer / Steps / Timeline` 等核心组件均有对应业务场景，极大减少样式开发量 |
| @ant-design/charts | 2.x | 基于 G2 的 React 图表库，与 Ant Design 风格统一，声明式 API，4 种图表满足所有统计需求 |
| Vite | 8.x | 毫秒级冷启动，HMR 极快；内置 `proxy` 配置解决开发跨域 |
| React Router | v7 | 声明式路由，支持嵌套布局（`<Layout>` + `<Outlet>`） |
| Axios | 1.x | HTTP 客户端，统一配置 `baseURL`，`.then(r => r.data)` 解包响应 |

### 3.2 后端技术栈

| 技术 | 版本 | 选型理由 |
|------|------|----------|
| FastAPI | 0.115 | 自动生成 OpenAPI 文档（`/docs`），原生集成 Pydantic 校验，异步性能优秀 |
| SQLAlchemy | 2.0 | Python 最成熟的 ORM；`Mapped` 类型注解语法简洁，级联操作、懒加载/急加载配置灵活 |
| Pydantic | v2 | 请求/响应数据校验；`model_config = {"from_attributes": True}` 直接从 ORM 对象序列化，无需手动转换 |
| Uvicorn | 0.30 | ASGI 服务器，支持热重载（`--reload`），生产可配合 Gunicorn 多进程 |

### 3.3 数据库

| 方案 | 说明 |
|------|------|
| SQLite 3 | 零配置文件级数据库，无需安装数据库服务；项目启动时自动建表 + 种子数据；数据持久化至 `backend/crm.db` |
| 可扩展性 | 仅需修改 `DATABASE_URL` 连接字符串即可切换至 PostgreSQL / MySQL，ORM 层无需改动 |

### 3.4 可视化方案

Dashboard 页面使用 4 种图表覆盖不同分析维度，均来自 `@ant-design/charts`，数据由后端 `/api/analytics/*` 系列接口实时计算返回：

| 图表 | API 端点 | SQL 逻辑 | 业务含义 |
|------|----------|----------|----------|
| 饼图 Pie | `/api/analytics/customer-industry` | `GROUP BY industry COUNT(*)` | 客户行业分布，了解客户构成 |
| 漏斗图 Funnel | `/api/analytics/sales-funnel` | `GROUP BY stage COUNT(*), SUM(amount)` | 销售阶段漏斗，识别转化瓶颈 |
| 折线图 Line | `/api/analytics/amount-trend` | `strftime('%Y-%m') + SUM(amount)` 近 6 个月 | 月度金额趋势，跟踪业绩走势 |
| 柱状图 Column | `/api/analytics/customer-growth` | `strftime('%Y-%m') + COUNT(*)` 近 6 个月 | 月度客户增长，评估获客效率 |

顶部 4 张统计卡片（客户总数 / 活跃机会数 / 本月新增客户 / 预估总金额）来自 `/api/analytics/overview`，提供一目了然的业务概览。

## 四、AI 使用方式

### 4.1 整体开发模式

本项目采用 **Claude Code CLI + superpowers 技能体系**，整个开发过程分三个阶段，每阶段有不同的 AI 使用场景和 Prompt 设计策略：

---

### 4.2 阶段一：需求探索（brainstorming skill）

**使用场景：** 从原始业务描述出发，通过问答式对话明确 MVP 边界和功能需求。

**用户输入 Prompt：**

> 我想构建一个面向企业销售场景的 CRM 系统，包含客户管理、销售流程管理、数据分析三个模块，用于管理客户信息、跟踪销售过程，并通过数据分析提升销售效率。请帮我分析需求，产出一份完整的需求规范文档。

**输出结构化需求规范** — 生成 `docs/superpowers/specs/2026-04-15-crm-requirements-spec.md`

**Prompt 设计要点：** 用户 Prompt 只需陈述业务目标，skill 负责追问细节、定义边界、补充验收标准。避免在初始 Prompt 中过度约束技术实现方案。

---

### 4.3 阶段二：技术规范生成（brainstorming skill）

**使用场景：** 将已确认的需求规范转化为可执行的技术设计文档和分阶段实现计划。

**用户输入 Prompt：**

> 需求规范已确认，请基于以下约束生成完整的系统技术设计方案：
> - 后端：FastAPI + SQLAlchemy 2.0 + Pydantic v2 + SQLite
> - 前端：React + TypeScript + Ant Design + Vite
> - 数据模型：客户 / 联系人 / 销售机会 / 跟进记录四个实体

**输出结构化技术设计** — 生成 `docs/superpowers/specs/2026-04-15-crm-technical-design.md`

**Prompt 设计要点：** 在此阶段的 Prompt 中明确声明技术栈版本和环境约束，这些约束会被嵌入所有后续的实现计划任务中，避免在执行阶段才发现兼容性问题。

---

### 4.4 阶段三：代码实现（writing-plans skill、subagent-driven-development skill）

**使用场景：** 以需求文档和技术设计文档为输入，按模块分步骤生成任务执行计划，由主 agent 派发独立 subagent 逐任务执行。

**用户输入 Prompt：**

> 技术设计已确认，现在需要完成后端核心 API 的实现任务，首先生成执行计划，我审阅后按计划生成代码。（后续任务依次按照这种方式执行）

**输出结构化实现计划** — 生成 `docs/superpowers/plans/2026-04-15-*.md`

**Prompt 设计要点：** 按模块分步骤进行代码实现，生成的plan文件中拆分为单一职责的细粒度任务，每个任务携带相关设计文档上下文并配置增量验证，同时预置幂等性保障和常见错误规避方案，确保 subagent 可独立、可靠地逐任务执行。

---

### 4.5 错误修正方式

开发过程中遇到的典型问题及修正策略：

| 问题现象 | 发现方式 | 修正策略 |
|----------|----------|----------|
| Write 工具单次内容过长触发参数截断 | 工具返回 `InputValidationError` | 将大文件（如含 30 个种子客户的 `seed.py`）拆为：先写骨架，再用 Edit 追加数据部分 |
| Pydantic v2 schema 生成报 `PydanticUserError` | 后端启动时立即抛出异常，日志显示 `orm_mode` 字段不合法 | AI 初次生成时沿用了 Pydantic v1 写法 `orm_mode = True`；v2 已改为 `model_config = {"from_attributes": True}`，逐一更新四个 schema 文件后恢复正常 |
| 删除客户后关联的联系人和机会记录未随之清除 | 手动测试删除操作，刷新列表后发现孤儿数据仍存在 | 检查 ORM 定义发现 `relationship` 缺少 `cascade="all, delete-orphan"` 配置；补充后重新验证，级联删除正常触发 |
| 种子数据每次重启服务都重复写入导致数据库膨胀 | 第二次启动后列表出现大量同名客户 | `seed.py` 初版没有幂等判断；修改为先检查 `db.query(Customer).count() == 0`，仅首次空库时执行写入，重启服务不再重复插入 |
| 前端开发时所有 API 请求返回 404 | 浏览器 Network 面板显示请求打到 `localhost:5173/api/...` 而非后端 | Vite `proxy` 配置含有多余的 `rewrite` 规则把 `/api` 前缀去掉了，而 FastAPI 路由本身已带 `/api` 前缀；删除 rewrite 后请求正常转发至 `localhost:8000` |
| `@ant-design/charts` 图表渲染空白且无报错 | Dashboard 页面图表区域空白，控制台无 Error | AI 生成时参照了 1.x 版本的 `config` 对象写法；2.x 已改为将配置项直接作为组件 prop 传入（如 `xField`、`yField`、`angleField`），按新 API 重写图表组件后正常渲染 |

## 五、核心流程说明

### 5.1 完整业务链路

```
录入客户 ──▶ 添加联系人 ──▶ 创建销售机会 ──▶ 记录跟进 ──▶ 推进阶段 ──▶ 赢单 / 输单
    │                                                                     │
    └────────────────── 数据看板实时反映业务状态 ◀────────────────────────────┘
```

### 5.2 流程详解

**第一步：录入客户**

销售人员在「客户管理」页面点击「新建客户」，右侧弹出 Drawer 抽屉表单，填写客户名称、行业、规模、来源、地区等信息。提交后前端调用 `POST /api/customers`，后端 Pydantic 校验通过后创建 Customer 记录并返回，列表自动刷新。支持按名称关键字搜索、按行业和状态下拉筛选、分页加载。

**第二步：添加联系人**

点击客户名称进入客户详情页，在「联系人」卡片中点击「新增」，Modal 表单填写姓名、职位、电话、邮箱，可勾选「主要联系人」。前端调用 `POST /api/contacts`，联系人列表即时刷新。支持编辑和删除。

**第三步：创建销售机会**

在「销售管理」页面点击「新建机会」，Drawer 表单通过下拉搜索关联到已录入客户，填写机会名称、预估金额、预计成交日期、优先级，初始阶段默认为「初步接触」。前端调用 `POST /api/opportunities`。

**第四步：记录跟进**

点击机会名称进入机会详情页，在「跟进记录」卡片中点击「添加跟进」，Modal 表单选择跟进方式（电话 / 邮件 / 拜访 / 会议 / 其他），填写跟进内容和下一步计划。前端调用 `POST /api/followups`，跟进记录以 Timeline 时间线形式展示，最新在上。

**第五步：推进销售阶段**

在机会详情页的「阶段进度」卡片中，Steps 组件展示完整销售漏斗：
初步接触 ──▶ 需求确认 ──▶ 方案报价 ──▶ 商务谈判 ──▶ 赢单 / 输单

点击目标阶段即可推进，前端调用 `PUT /api/opportunities/{id}` 更新 `stage` 字段。阶段进入「赢单」或「输单」后，该机会将从活跃机会统计中排除。

**第六步：数据看板**

所有操作实时反映在 Dashboard 数据看板：

- **统计卡片**：客户总数 / 活跃机会数 / 本月新增客户 / 预估总金额（万元）
- **行业分布饼图**：`GROUP BY industry` 统计客户数，直观展示客户结构
- **销售漏斗图**：按阶段统计机会数量，识别转化率低的卡点
- **金额趋势折线图**：按月汇总机会金额，展示近 6 个月业绩走势
- **客户增长柱状图**：按月统计新增客户数，评估获客效率
